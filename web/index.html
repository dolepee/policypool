Hook mechanism

Small policy surface. Real swap enforcement.

No oracle, no offchain matcher, no governance layer. The v1 Hook only checks max input and rolling daily volume, then either lets the v4 swap continue or refuses it before execution. 

if amountIn > maxSwapAmount:
  revert PolicyBlocked("MAX_SWAP_EXCEEDED")

if spentToday + amountIn > dailyCap:
  revert PolicyBlocked("DAILY_CAP_EXCEEDED")

spentToday += amountIn
emit SwapAccepted(poolId, trader, amountIn)

Reviewer checks

The refusal is enforced by the Hook, not by the UI.

Hook-level block

beforeSwap reverts before the pool consumes liquidity. The router only records the caught failure.

No router bypass

Any v4 swap against these pools has to pass the same Hook callback through PoolManager.

Mock assets by design

The proof isolates Hook behavior from token liquidity noise while still using live X Layer transactions.

Not dynamic fees

The Hook does not reprice the trade. It changes the pool's execution boundary: accept or refuse.

Market path

Bounded flow for LPs who cannot watch every block.

PolicyPool is a second pool class, not a replacement for permissionless pools. It gives launch teams, treasuries, and market makers a public flow envelope the pool enforces itself.

01 New asset launches

Cap early whale flow while liquidity is shallow, without hiding the rule in a private backend.

02 Treasury liquidity

Publish max-swap and daily-volume limits so the operating envelope is visible before execution.

03 Market makers

Run strict and loose pools side by side for the same pair, each with its own enforceable covenant.
